1. Infraestrutura e Docker

  ------------------- --------- ------------------------------------------------------------------------------------------------------------------------------------
  Docker e Ambiente             

  ☐                   CRÍTICO   docker-compose.yml define todos os 7 serviços: nginx, php-fpm, reverb, horizon, scheduler, postgres, redis — sem serviços faltando

  ☐                   CRÍTICO   Nenhuma credencial hardcoded — tudo via variáveis de ambiente (.env) — verificar com git grep 'password' src/

  ☐                   CRÍTICO   .env.example existe e contém TODAS as variáveis necessárias com valores de exemplo

  ☐                   CRÍTICO   .gitignore inclui .env, storage/logs, vendor, node_modules

  ☐                   ALTO      Health checks nos serviços do docker-compose (postgres, redis) — --health-cmd pg_isready

  ☐                   ALTO      Makefile com comandos: make dev, make stop, make shell-php, make shell-vue, make test

  ☐                   ALTO      Volumes persistentes para postgres e redis (dados sobrevivem ao docker compose down)

  ☐                   ALTO      Nginx configurado com gzip, cache de assets estáticos, proxy para php-fpm e reverb

  ☐                   MÉDIO     Dockerfiles com multi-stage build (build stage separado do runtime)

  ☐                   MÉDIO     Redis configurado com maxmemory e allkeys-lru policy no redis.conf

  ☐                   MÉDIO     PHP-FPM configurado com pool adequado (pm.max_children ajustado)

2. Backend Laravel — Arquitetura

  ------------------------- --------- ------------------------------------------------------------------------------------------------------------------------------------------------------------
  Estrutura e Organização             

  ☐                         CRÍTICO   Toda a lógica de negócio está nos Services, nunca nos Controllers — Controller: recebe request, chama service, retorna resource

  ☐                         CRÍTICO   Repositórios implementam interfaces (não usar Eloquent diretamente no Service) — EloquentStudySessionRepository implements StudySessionRepositoryInterface

  ☐                         CRÍTICO   Injeção de dependência via type hints nos construtores (não new Class() dentro de métodos)

  ☐                         ALTO      DTOs imutáveis com readonly properties para transporte de dados validados entre camadas

  ☐                         ALTO      API versionada: prefix /api/v1 em todas as rotas, grupos nomeados

  ☐                         ALTO      Namespace de módulos: app/Modules/{Auth,StudySessions,Technologies,Analytics}/

  ☐                         ALTO      HasApiResponse trait centraliza formatação de respostas de sucesso e erro

  ☐                         ALTO      Exceções de domínio customizadas (ConcurrentSessionException, etc.) com código semântico

  ☐                         MÉDIO     BaseModel.php com UUID, serializeDate() e $guarded vazio (usar $fillable)

  ☐                         MÉDIO     AppServiceProvider registra binds interface → implementação concreta

  ☐                         MÉDIO     Nenhuma query N+1: usar eager loading with('technology') em listagens de sessões

  ☐                         BAIXO     PHPDoc nas classes públicas: @param, @return, @throws documentados

  --------------------------- --------- -----------------------------------------------------------------------------------------------------------
  API e Tratamento de Erros             

  ☐                           CRÍTICO   Handler.php renderiza todos os erros como JSON com envelope {success:false, error:{code,message,details}}

  ☐                           CRÍTICO   Respostas 401 NUNCA revelam se o email existe ou não (mensagem genérica) — prevenção de user enumeration

  ☐                           CRÍTICO   Respostas 403 em tentativa de acesso cross-user (não 404 — info leak)

  ☐                           CRÍTICO   Form Requests validam propriedade do technology_id (pertence ao usuário autenticado)

  ☐                           ALTO      EnsureJsonResponse middleware garante Content-Type: application/json em todos os erros

  ☐                           ALTO      Rate limiting: login 3/min IP, register 5/min IP; leituras 60/min; escrita geral 30/min; ver AppServiceProvider + api.php + throttle.sliding em sessões

  ☐                           ALTO      Todos os endpoints retornam o mesmo envelope de sucesso {success:true, data:{...}}

  ☐                           ALTO      Paginação consistente: ?page=1&per_page=15 com meta:{total,per_page,current_page,last_page}

  ☐                           ALTO      Resposta 202 Accepted para POST /api/v1/analytics/recalculate (operação assíncrona)

  ☐                           MÉDIO     Códigos de erro semânticos em SNAKE_UPPER: VALIDATION_ERROR, CONCURRENT_SESSION, NOT_FOUND

  ☐                           MÉDIO     Logs estruturados: contexto userId e requestId em todos os erros de domínio

  ☐                           BAIXO     GET /api/health (e opcionalmente GET /health web) valida PostgreSQL, Redis, fila e Reverb
  --------------------------- --------- -----------------------------------------------------------------------------------------------------------

  ------------- --------- -----------------------------------------------------------------------------------------------
  Cache Redis             

  ☐             CRÍTICO   Cache NUNCA aplicado no Controller — apenas no Service via Cache::tags()->remember()

  ☐             CRÍTICO   Cache Tags usados em TODOS os puts: sem tags, flush é impossível sem conhecer todas as chaves

  ☐             CRÍTICO   RecalculateMetricsJob invalida via tag flush APÓS o UPSERT, nunca antes

3. Eventos, Jobs e WebSocket

  --------------------------- --------- -------------------------------------------------------------------------------------------------------
  Event-Driven Architecture             

  ☐                           CRÍTICO   Controllers NUNCA disparam jobs diretamente — apenas via event() após o Service persistir

  ☐                           CRÍTICO   Listeners são burros: apenas limpam cache, despacham jobs ou emitem broadcast — sem lógica de negócio

  ☐                           CRÍTICO   StudySessionDeleted carrega value objects, não Eloquent model (modelo já deletado ao disparar)

  ☐                           ALTO      EventServiceProvider com lista explícita de listeners (sem autodiscovery — performance)

  ☐                           ALTO      Ordem dos listeners importa: InvalidateSessionCache ANTES de DispatchMetricsRecalculation

  ☐                           ALTO      RecalculateMetricsJob implementa ShouldBeUnique com uniqueId() = userId

  ☐                           ALTO      $tries=3, $backoff=[30,60,120], $timeout=30 configurados em todos os jobs críticos

  ☐                           ALTO      failed() method em todos os jobs: Log::error com contexto completo (userId, attempt, trace)

  ☐                           MÉDIO     ->delay(now()->addSeconds(2)) no dispatch do job: agrupa múltiplos eventos rápidos

  ☐                           MÉDIO     Horizon configurado com supervisores por fila: fila 'metrics' com 2-8 workers auto-scale

  ☐                           MÉDIO     GenerateWeeklySummaryJob agendado: ->weeklyOn(1, '03:00')->withoutOverlapping()->onOneServer()

  ☐                           BAIXO     Horizon Dashboard protegido por gate: apenas admin acessa /horizon em produção
  --------------------------- --------- -------------------------------------------------------------------------------------------------------

  ---------------------------- --------- ------------------------------------------------------------------------------------------------
  WebSocket (Laravel Reverb)             

  ☐                            CRÍTICO   Canal privado dashboard.{userId}: usuário só recebe seus próprios eventos

  ☐                            CRÍTICO   routes/channels.php autoriza canal: (string)$user->id === (string)$userId

  ☐                            ALTO      broadcastAs() com ponto no início: '.metrics.updated' (sem prefixo App\\Events automático)

  ☐                            ALTO      broadcastWith() retorna payload mínimo — não serializar o model Eloquent inteiro

  ☐                            ALTO      Sequência correta: .metrics.recalculating (spinner) → job roda → .metrics.updated (dados)

  ☐                            ALTO      elapsed_seconds calculado no backend (NOW() - started_at) e enviado no evento .session.started

  ☐                            MÉDIO     Frontend desconecta WebSocket no onUnmounted (evita memory leak entre navegações)

  ☐                            MÉDIO     RealtimeBadge mostra status da conexão: verde (conectado), vermelho (desconectado)

  ☐                            MÉDIO     Polling de fallback (2 minutos) quando WebSocket está desconectado

  ☐                            BAIXO     Reconexão automática gerenciada pelo Pusher/Echo (sem implementação manual)
