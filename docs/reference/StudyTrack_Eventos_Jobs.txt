+----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| POR QUE EVENT-DRIVEN NESTE SISTEMA?                                                                                                                                                                                                                                                          |
+----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| Sem eventos, um Controller de criação de sessão faria: salvar no banco, recalcular métricas, invalidar cache, enviar WebSocket. Isso é acoplamento direto entre camadas — o Controller precisa conhecer tudo. Com o custo adicional de que, se o recalculo falhar, a requisição falha junto. |
|                                                                                                                                                                                                                                                                                              |
| Com eventos: o Controller salva a sessão e dispara StudySessionCreated. Pronto. Listeners cuidam do resto de forma assíncrona, em fila, com retry automático. A resposta HTTP chega em 5ms; o recalculo acontece em segundo plano.                                                           |
|                                                                                                                                                                                                                                                                                              |
| Princípio: O produtor do evento (Service/Controller) nunca conhece os consumidores (Listeners/Jobs). Adicionar um novo comportamento ao criar sessão é adicionar um Listener — sem tocar no código existente. Open/Closed Principle na prática.                                              |

2. EventServiceProvider — Registro Completo

O EventServiceProvider mapeia cada evento aos seus listeners. A ordem dos listeners dentro de um array importa — eles são executados em sequência, sincronamente, no mesmo request. Por isso a ordem: invalidação de cache primeiro, depois disparo do job assíncrono.

4. Implementação dos Listeners

4.1 InvalidateSessionCache

Primeiro listener a executar. Remove o cache de sessão ativa do usuário. Deve ser síncrono e completar antes do job ser despachado.

+--------------------------------------------------------------------------------------------------+
| <?php // app/Listeners/StudySession/InvalidateSessionCache.php                                   |
|                                                                                                  |
| class InvalidateSessionCache                                                                     |
|                                                                                                  |
| {                                                                                                |
|                                                                                                  |
| public function handle(StudySessionCreated|StudySessionUpdated|StudySessionDeleted $event): void |
|                                                                                                  |
| {                                                                                                |
|                                                                                                  |
| $userId = match(true) {                                                                          |
|                                                                                                  |
| $event instanceof StudySessionDeleted => $event->userId,                                         |
|                                                                                                  |
| default => $event->session->user_id,                                                             |
|                                                                                                  |
| };                                                                                               |
|                                                                                                  |
| // Invalida cache de sessão ativa e qualquer lista de sessões cacheada                           |

| Log::debug('Session cache invalidated', ['userId' => $userId]);                                  |
|                                                                                                  |
| }                                                                                                |
|                                                                                                  |
| }                                                                                                |

4.2 DispatchMetricsRecalculation

Analisa quais campos mudaram e decide se precisa de recalculo completo ou parcial antes de despachar o job.

+--------------------------------------------------------------------------------------------------+
| <?php // app/Listeners/StudySession/DispatchMetricsRecalculation.php                             |
|                                                                                                  |
| class DispatchMetricsRecalculation                                                               |
|                                                                                                  |
| {                                                                                                |
|                                                                                                  |
| public function handle(StudySessionCreated|StudySessionUpdated|StudySessionDeleted $event): void |
|                                                                                                  |
| {                                                                                                |
|                                                                                                  |
| [$userId, $fullRecalc] = match(true) {                                                           |
|                                                                                                  |
| // Deleted: sempre recalculo completo (totais mudaram)                                           |
|                                                                                                  |
| $event instanceof StudySessionDeleted => [$event->userId, true],                                 |
|                                                                                                  |
| // Updated: fullRecalc apenas se dados de tempo/tecnologia mudaram                               |
|                                                                                                  |
| $event instanceof StudySessionUpdated => [                                                       |
|                                                                                                  |
| $event->session->user_id,                                                                        |
|                                                                                                  |
| !empty(array_intersect(                                                                          |
|                                                                                                  |
| array_keys($event->changedFields),                                                               |
|                                                                                                  |
| ['started_at', 'ended_at', 'technology_id', 'duration_min']                                      |
|                                                                                                  |
| )),                                                                                              |
|                                                                                                  |
| ],                                                                                               |
|                                                                                                  |
| // Created: sempre fullRecalc                                                                    |
|                                                                                                  |
| default => [$event->session->user_id, true],                                                     |
|                                                                                                  |
| };                                                                                               |
|                                                                                                  |
| RecalculateMetricsJob::dispatch($userId, $fullRecalc)                                            |
|                                                                                                  |
| ->onQueue('metrics')                                                                             |
|                                                                                                  |
| ->delay(now()->addSeconds(2)); // pequeno delay para agrupar eventos rápidos                     |
|                                                                                                  |
| }                                                                                                |
|                                                                                                  |
| }                                                                                                |

| <?php // app/Jobs/RecalculateMetricsJob.php                                       |
|                                                                                   |
| use Illuminate\Contracts\Queue\ShouldBeUnique;                                    |
|                                                                                   |
| class RecalculateMetricsJob implements ShouldQueue, ShouldBeUnique                |
|                                                                                   |
| {                                                                                 |
|                                                                                   |
| use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;                |
|                                                                                   |
| public int $tries = 3;                                                            |
|                                                                                   |
| public array $backoff = [30, 60, 120];                                            |
|                                                                                   |
| public int $timeout = 30;                                                         |
|                                                                                   |
| public string $queue = 'metrics';                                                 |
|                                                                                   |
| // ShouldBeUnique: previne jobs duplicados por userId                             |
|                                                                                   |
| public int $uniqueFor = 5; // segundos que o lock é mantido                       |
|                                                                                   |
| public function uniqueId(): string { return $this->userId; }                      |
|                                                                                   |
| public function __construct(                                                      |
|                                                                                   |
| public readonly string $userId,                                                   |
|                                                                                   |
| public readonly bool $fullRecalc = true,                                          |
|                                                                                   |
| ) {}                                                                              |
|                                                                                   |
| public function handle(                                                           |
|                                                                                   |
| UserMetricsAggregator $userAgg,                                                   |
|                                                                                   |
| TechMetricsAggregator $techAgg,                                                   |
|                                                                                   |
| DailyMinutesAggregator $dailyAgg,                                                 |
|                                                                                   |
| AnalyticsService $analyticsService,                                               |
|                                                                                   |
| ): void {                                                                         |
|                                                                                   |
| // 1. Recalcula no banco em uma única transação atômica                           |
|                                                                                   |
| DB::transaction(function () use ($userAgg, $techAgg, $dailyAgg) {                 |
|                                                                                   |
| $userAgg->recalculate($this->userId);                                             |
|                                                                                   |
| $techAgg->recalculate($this->userId);                                             |
|                                                                                   |
| $dailyAgg->recalculate($this->userId);                                            |
|                                                                                   |
| });                                                                               |
|                                                                                   |
| // 2. Invalida cache desatualizado                                                |
|                                                                                   |
| Cache::tags(['analytics', "user:{$this->userId}"])->flush();                      |
|                                                                                   |
| // 3. Monta payload e faz cache warming (pré-aquece para próximo request)         |
|                                                                                   |
| $freshData = $analyticsService->buildDashboardPayload($this->userId);             |
|                                                                                   |
| Cache::tags(['analytics', "user:{$this->userId}"])                                |
|                                                                                   |
| ->put("dashboard:{$this->userId}", $freshData, now()->addMinutes(5));             |
|                                                                                   |
| // 4. Dispara evento que notifica frontend via WebSocket                          |
|                                                                                   |
| event(new MetricsRecalculated($this->userId, $freshData));                        |
|                                                                                   |
| }                                                                                 |
|                                                                                   |
| public function failed(\Throwable $e): void                                       |
|                                                                                   |
| {                                                                                 |
|                                                                                   |
| Log::error('RecalculateMetricsJob failed', [                                      |
|                                                                                   |
| 'userId' => $this->userId,                                                        |
|                                                                                   |
| 'attempt' => $this->attempts(),                                                   |
|                                                                                   |
| 'error' => $e->getMessage(),                                                      |
|                                                                                   |
| 'fullTrace' => $e->getTraceAsString(),                                            |
|                                                                                   |
| ]);                                                                               |
|                                                                                   |
| // Notificar frontend que o spinner pode ser removido (falhou, mas não travar UI) |
|                                                                                   |
| broadcast(new MetricsRecalculating($this->userId))->via('reverb');                |
|                                                                                   |
| }                                                                                 |
|                                                                                   |
| }                                                                                 |

| public array $backoff = [300];                                        |
|                                                                       |
| public int $timeout = 300;                                            |
|                                                                       |
| public function handle(WeeklySummaryAggregator $aggregator): void     |
|                                                                       |
| {                                                                     |
|                                                                       |
| $processed = 0;                                                       |
|                                                                       |
| $cutoff = now()->subDays(30);                                         |
|                                                                       |
| // Processa apenas usuários com sessões nos últimos 30 dias           |
|                                                                       |
| // chunk(100): carrega 100 usuários por vez na memória                |
|                                                                       |
| User::whereHas('studySessions', function ($q) use ($cutoff) {         |
|                                                                       |
| $q->where('started_at', '>=', $cutoff);                               |
|                                                                       |
| })->chunk(100, function ($users) use ($aggregator, &$processed) {     |
|                                                                       |
| foreach ($users as $user) {                                           |
|                                                                       |
| $aggregator->recalculateCurrentWeek($user->id);                       |
|                                                                       |
| Cache::tags(['analytics', "user:{$user->id}", 'weekly'])->flush();    |
|                                                                       |
| $processed++;                                                         |
|                                                                       |
| }                                                                     |
|                                                                       |
| });                                                                   |
|                                                                       |
| Log::info('WeeklySummary generated', [                                |
|                                                                       |
| 'users_processed' => $processed,                                      |
|                                                                       |
| 'duration_ms' => $this->job?->getTimeout(),                           |
|                                                                       |
| ]);                                                                   |
|                                                                       |
| }                                                                     |
|                                                                       |
| }                                                                     |

| // routes/console.php — Agendamento                                   |
|                                                                       |
| Schedule::job(new GenerateWeeklySummaryJob())                         |
|                                                                       |
| ->weeklyOn(1, '03:00') // 1 = segunda-feira                           |
|                                                                       |
| ->name('weekly-summary')                                              |
|                                                                       |
| ->withoutOverlapping() // previne execuções simultâneas               |
|                                                                       |

7.2 Cenário: Job Falha na Primeira Tentativa

+------------------------------------------------------------------------------+
| # RETRY COM BACKOFF EXPONENCIAL                                              |
|                                                                              |
| t=0s RecalculateMetricsJob tentativa 1 iniciada                              |
|                                                                              |
| → DB::transaction falha (ex: lock timeout no PostgreSQL)                     |
|                                                                              |
| → Throwable capturado pelo Horizon                                           |
|                                                                              |
| → Job marcado como 'released' (não 'failed' ainda)                           |
|                                                                              |
| → $backoff[0] = 30s: job fica bloqueado por 30 segundos                      |
|                                                                              |
| t=30s RecalculateMetricsJob tentativa 2                                      |
|                                                                              |
| → Sucesso ✓                                                                  |
|                                                                              |
| → Fluxo normal: UPSERT → cache flush → warm → MetricsRecalculated            |
|                                                                              |
| # SE tentativa 2 TAMBÉM falhar:                                              |
|                                                                              |
| t=90s RecalculateMetricsJob tentativa 3 (30s + 60s de backoff)               |
|                                                                              |
| → Falha novamente ✗                                                          |
|                                                                              |
| → $tries esgotados: job movido para failed_jobs table                        |
|                                                                              |
| → RecalculateMetricsJob::failed(Throwable) chamado                           |
|                                                                              |
| → Log::error com stack trace completo                                        |
|                                                                              |
| → broadcast(MetricsRecalculating) para remover spinner no frontend           |
|                                                                              |
| → Horizon Dashboard mostra job como FAILED (visível em /horizon)             |
|                                                                              |
| # IMPACTO NO USUÁRIO quando job falha completamente:                         |
|                                                                              |
| # → Dashboard mostra dados do último recalculo bem-sucedido (do cache/banco) |
|                                                                              |
| # → Spinner some (frontend não fica travado)                                 |
|                                                                              |
| # → Próxima ação do usuário (nova sessão) dispara novo job                   |
|                                                                              |
| # → Sistema se auto-recupera sem intervenção manual                          |

7.3 Cenário: Dois Jobs do Mesmo Usuário Simultâneos (ShouldBeUnique)

+-------------------------------------------------------------------------------------+
| # ShouldBeUnique previne processamento duplo                                        |
|                                                                                     |
| t=0s Usuário cria 3 sessões em rápida sequência (ex: corrigindo erros)              |
|                                                                                     |
| → 3x StudySessionCreated events disparados                                          |
|                                                                                     |
| → 3x DispatchMetricsRecalculation::handle() executa                                 |
|                                                                                     |
| → 3x RecalculateMetricsJob::dispatch($userId) chamado                               |
|                                                                                     |
| t=1s Job 1: Redis adquire lock 'unique:RecalculateMetricsJob:{userId}' (5s TTL)     |
|                                                                                     |
| Job 1: entra na fila e começa a processar                                           |
|                                                                                     |
| t=1s Job 2: tenta adquirir lock → LOCK EXISTS → job DESCARTADO (não re-enfileirado) |
|                                                                                     |
| t=1s Job 3: tenta adquirir lock → LOCK EXISTS → job DESCARTADO                      |
|                                                                                     |
| t=3s Job 1 conclui: recalcula dados das 3 sessões de uma vez                        |
|                                                                                     |
| → UPSERT calcula totais corretos incluindo as 3 sessões                             |
|                                                                                     |
| → Lock liberado                                                                     |
|                                                                                     |
| # RESULTADO: 1 recalculo acontece (não 3), com dados finais corretos                |
|                                                                                     |
| # SEM ShouldBeUnique: 3 recalculos simultâneos, race conditions nos UPSERTs         |

|                                                                       |
| });                                                                   |
|                                                                       |
| }                                                                     |
|                                                                       |
| function disconnect(): void {                                         |
|                                                                       |
| echo?.disconnect();                                                   |
|                                                                       |
| echo = null;                                                          |
|                                                                       |
| isConnected.value = false;                                            |
|                                                                       |

|                                                                       |

11. Checklist de Implementação

Eventos e Listeners

-   [ ] EventServiceProvider com mapeamento explícito de todos os eventos

-   [ ] 3 eventos de sessão: Created, Updated, Deleted

-   [ ] 1 evento analytics: MetricsRecalculated

-   [ ] 3 broadcast events: MetricsUpdated, MetricsRecalculating, SessionStarted

-   [ ] Listeners de sessão: InvalidateSessionCache, DispatchMetricsRecalculation

-   [ ] Listeners de analytics: UpdateCacheWithFreshData, BroadcastMetricsUpdate

-   [ ] StudySessionDeleted usa value objects (não Eloquent model) no payload

Jobs e Horizon

-   [ ] RecalculateMetricsJob implementa ShouldBeUnique (uniqueId = userId)

-   [ ] $tries=3, $backoff=[30,60,120], $timeout=30 configurados

-   [ ] failed() method com Log::error e broadcast de remoção do spinner

-   [ ] GenerateWeeklySummaryJob agendado via Schedule::job()->weeklyOn(1, '03:00')

-   [ ] ->withoutOverlapping() e ->onOneServer() no agendamento

-   [ ] config/horizon.php com supervisores para filas 'metrics' e 'default'

-   [ ] Horizon Dashboard protegido por gate (apenas admin em produção)

WebSocket e Frontend

-   [ ] routes/channels.php com autorização do canal dashboard.{userId}

-   [ ] Broadcast events com broadcastAs() usando ponto para evitar prefixo

-   [ ] broadcastWith() retornando payload mínimo necessário

-   [ ] useWebSocket composable com connect/disconnect e estado de conexão

-   [ ] DashboardView chama connect() em onMounted e disconnect() em onUnmounted

-   [ ] analyticsStore.updateFromWebSocket() atualiza sem request HTTP

-   [ ] RealtimeBadge indica status da conexão WebSocket ao usuário

+----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| ARMADILHA: LISTENER SINCRONO QUE FAZ I/O LENTO                                                                                                                                                                 |
+----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| Problema: Listeners registrados em EventServiceProvider executam sincronamente no mesmo thread do request. Se um listener fizer I/O lento (query pesada, chamada HTTP externa), o response HTTP vai atrasar.   |
|                                                                                                                                                                                                                |
| Regra: Listeners NUNCA fazem queries pesadas. Eles apenas: (1) limpam cache com flush, (2) despacham jobs para a fila, (3) fazem broadcasts simples. Todo processamento pesado vai para o Job.                 |
|                                                                                                                                                                                                                |
| Solução alternativa: Implementar ShouldQueue no Listener: o listener em si vira um job assíncrono. Mas para este sistema, a abordagem atual (Listener síncrono leve + Job pesado) é mais explícita e testável. |
