1. Visão Geral da Estratégia

A estratégia de cache do StudyTrack é organizada em quatro camadas independentes, cada uma com responsabilidades e TTLs distintos. Cada camada usa um banco Redis diferente para isolar responsabilidades operacionais:

  ---------------------------------------------------------------------------------------------------------------------
  DB     Camada                  TTL máx      Padrão          Dados cacheados
  ------ ----------------------- ------------ --------------- ---------------------------------------------------------
  DB 1   Analytics / Dashboard   5–15 min     Cache-Aside     Payload do dashboard, métricas, série temporal, heatmap

  DB 2   Dados operacionais      1h           Cache-Aside     Tecnologias do usuário, perfil, busca por trigrama

  DB 3   Sessões de aplicação    7 dias       Write-Through   Tokens Sanctum, sessão ativa do usuário

  DB 4   Filas (Queue)           Permanente   Queue Backend   Jobs RecalculateMetrics, BroadcastUpdate etc.
  ---------------------------------------------------------------------------------------------------------------------

1.1 Padrão Cache-Aside (Lazy Loading)

O padrão principal utilizado no sistema. A aplicação tenta o cache primeiro; em caso de miss, busca no banco, armazena no cache e retorna. O banco é a fonte da verdade — o cache é apenas uma cópia acelerada.

2. Cache Tags — O Mecanismo Central

Cache Tags é a funcionalidade mais importante para este sistema. Permite associar múltiplas chaves de cache a uma ou mais "tags" e depois invalidar todas elas com uma única chamada. Isso resolve o problema de invalidar dados de um usuário específico sem afetar outros.

+-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| POR QUE CACHE TAGS E NÃO CHAVES INDIVIDUAIS?                                                                                                                                              |
+-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| Imagine um usuário que acessou o dashboard com parâmetros diferentes:                                                                                                                     |
|                                                                                                                                                                                           |
| -   dashboard:uuid-123                                                                                                                                                                    |
|                                                                                                                                                                                           |
| -   timeseries:uuid-123:30d                                                                                                                                                               |
|                                                                                                                                                                                           |
| -   timeseries:uuid-123:90d                                                                                                                                                               |
|                                                                                                                                                                                           |
| -   timeseries:uuid-123:custom:...                                                                                                                                                        |
|                                                                                                                                                                                           |
| -   tech-stats:uuid-123:sort:total_min                                                                                                                                                    |
|                                                                                                                                                                                           |
| -   heatmap:uuid-123:2025                                                                                                                                                                 |
|                                                                                                                                                                                           |
| Quando as métricas são recalculadas, todas essas chaves ficam desatualizadas. Com Tags:                                                                                                   |
|                                                                                                                                                                                           |
| -   Cache::tags(['analytics', 'user:uuid-123'])->flush();                                                                                                                                 |
|                                                                                                                                                                                           |
| Uma linha de código invalida todas. Com chaves individuais, você precisaria de 6+ chamadas e teria que manter uma lista de todas as chaves possíveis — inviável com parâmetros dinâmicos. |
+-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+

3. TTL por Camada de Dado

O TTL (Time To Live) de cada chave é definido com base na frequência de atualização do dado e no impacto de servir dados levemente desatualizados. Dados mais voláteis têm TTL menor; dados mais estáveis podem ser cacheados por mais tempo.

Classificação por Temperatura de Dado:

  ----------- --------- ---------------------------------------------------------------------------------------------------------------------------------------------------------
  QUENTE      2–5 min   Dados que mudam freqüentemente e são lidos com alta frequência. Sessão ativa, métricas do dashboard. Stale data por >5min seria perceptivel ao usuário.

  ----------- --------- ---------------------------------------------------------------------------------------------------------------------------------------------------------

  ----------- ----------- ------------------------------------------------------------------------------------------------------------------------------------------------------
  MORNO       10–15 min   Dados que mudam com eventos específicos (nova sessão) mas não continuamente. Série temporal, resumos semanais. Lag de 15min não afeta a experiência.

  ----------- ----------- ------------------------------------------------------------------------------------------------------------------------------------------------------

  ----------- -------- ---------------------------------------------------------------------------------------------------------------------------------------------------
  FRIO        1 hora   Dados estáveis que mudam apenas em ações explícitas do usuário. Lista de tecnologias, perfil, heatmap anual. Invalidados por evento, não por TTL.

  ----------- -------- ---------------------------------------------------------------------------------------------------------------------------------------------------

  ----------- -------- -------------------------------------------------------------------------------------------------------------------------------
  GELADO      7 dias   Dados de sessão e autenticação. Tokens Sanctum, sessões da aplicação. Gerenciados pelo middleware, não pelo AnalyticsService.

  ----------- -------- -------------------------------------------------------------------------------------------------------------------------------

  ------------------------------------------------------------------------------------------
  Chave de Cache                   TTL      Temp.    Evento de Invalidação
  -------------------------------- -------- -------- ---------------------------------------
  dashboard:{userId}               5 min    Quente   RecalculateMetricsJob finalizado

  user-metrics:{userId}            5 min    Quente   RecalculateMetricsJob finalizado

  tech-stats:{userId}:{sort}       5 min    Quente   RecalculateMetricsJob finalizado

  active-session:{userId}          2 min    Quente   POST /api/v1/study-sessions/start, PATCH …/end

  user-profile:{userId}            5 min    Quente   PUT /api/v1/auth/me

  timeseries:{userId}:{period}     15 min   Morno    RecalculateMetricsJob finalizado

  weekly:{userId}:{weeks}          15 min   Morno    RecalculateMetricsJob finalizado

  tech-search:{userId}:{q}         5 min    Morno    POST/PUT/DELETE /api/v1/technologies

  technologies:{userId}:{params}   1 hora   Frio     POST/PUT/DELETE /api/v1/technologies

  technology:{id}:{userId}         1 hora   Frio     PUT/DELETE /api/v1/technologies/{id}

  heatmap:{userId}:{year}          1 hora   Frio     RecalculateMetricsJob (semanal)

  rate-limit:auth:{ip}             1 min    Quente   Expiração automática (sliding window)

4. Pipeline de Invalidação de Cache

A invalidação é o problema mais difícil em caching. O sistema usa Event-Driven invalidation: eventos de domínio disparam listeners que invalidam as tags corretas. A aplicação nunca invalida cache diretamente no Controller — isso é responsabilidade dos Listeners.

5. Cache Warming e Cache Stampede

5.1 Cache Warming (Pré-aquecimento)

O passo 3 do RecalculateMetricsJob ilustra cache warming: após invalidar e recalcular, o job imediatamente popula o cache com os dados frescos. Isso garante que o próximo request do dashboard seja um HIT, mesmo que o WebSocket chegue antes do usuário navegar até o dashboard.

+-----------------------------------------------------------------------------------------------------------------------+
| QUANDO USAR CACHE WARMING?                                                                                            |
+-----------------------------------------------------------------------------------------------------------------------+
| -   Após recalculo de métricas: O job já tem os dados — é gratuíto populá-los no cache antes de terminar.             |
|                                                                                                                       |
| -   No login do usuário: AuthService pode pre-warm dashboard cache após autenticação. Primeira visita será HIT.       |
|                                                                                                                       |
| -   Agendamento noturno: Job agendado às 3h da manhã pode warm heatmap e weekly para usuários ativos. TTL até amanhã. |
|                                                                                                                       |
| -   NÃO usar para: Dados que mudam com alta frequência sem evento previsível. Cache-aside é suficiente.               |
+-----------------------------------------------------------------------------------------------------------------------+

5.2 Cache Stampede (Thundering Herd)

Cache stampede ocorre quando o TTL de uma chave expira e múltiplos workers tentam regenerar o cache simultaneamente, sobrecarregando o banco. Em produção com muitos usuários, isso é crítico. Para um portfólio com poucos usuários, o risco é baixo, mas a solução deve ser documentada.

+-------------------------------------------------------------------------------------+
| // Solução: Probabilistic Early Expiration (PER) + Lock                             |
|                                                                                     |
| // Em vez de esperar o TTL zerar, regenera alguns segundos antes (aleatório)        |
|                                                                                     |
| class AnalyticsService                                                              |
|                                                                                     |
| {                                                                                   |
|                                                                                     |
| public function getDashboardData(string $userId): array                             |
|                                                                                     |
| {                                                                                   |
|                                                                                     |
| $cacheKey = "dashboard:{$userId}";                                                  |
|                                                                                     |

|                                                                                     |

|                                                                                     |

|                                                                                     |

|                                                                                     |

|                                                                                     |

|                                                                                     |

|                                                                                     |

7. Cache de Sessões e Tokens

7.1 Tokens Sanctum no Redis

Por padrão, o Laravel Sanctum persiste tokens no PostgreSQL (tabela personal_access_tokens). Para melhorar a performance de autenticação, os tokens verificados recentemente são cacheados no Redis para evitar a query ao banco em cada request.

9. Cache no Frontend (Vue 3 + Pinia)

O frontend tem sua própria camada de cache em memória via Pinia stores. Isso evita refetch da API quando o usuário navega entre páginas sem que os dados tenham mudado. O WebSocket é o mecanismo de atualização em vez de polling ou refetch.

9.1 Analytics Store com Cache de Estado

+---------------------------------------------------------------------------------------------+
| // stores/analytics.store.ts                                                                |
|                                                                                             |
| export const useAnalyticsStore = defineStore('analytics', () => {                           |
|                                                                                             |
| // Estado: null = não carregado, objeto = carregado                                         |
|                                                                                             |
| const dashboard = ref<DashboardData | null>(null);                                          |
|                                                                                             |
| const lastFetchAt = ref<Date | null>(null);                                                 |
|                                                                                             |
| const isStale = ref(false); // true quando WebSocket sinalizou atualização                  |
|                                                                                             |
| const isLoading = ref(false);                                                               |
|                                                                                             |
| // TTL do cache em memória: 5 minutos (igual ao backend)                                    |
|                                                                                             |
| const STALE_AFTER_MS = 5 * 60 * 1000;                                                       |
|                                                                                             |
| // Getter: dados estão frescos?                                                             |
|                                                                                             |
| const isFresh = computed(() => {                                                            |
|                                                                                             |
| if (!lastFetchAt.value) return false;                                                       |
|                                                                                             |
| if (isStale.value) return false;                                                            |
|                                                                                             |
| return Date.now() - lastFetchAt.value.getTime() < STALE_AFTER_MS;                           |
|                                                                                             |
| });                                                                                         |
|                                                                                             |
| // Action: busca se necessário (cache-aside no frontend)                                    |
|                                                                                             |
| async function fetchDashboard(force = false): Promise<void> {                               |

|                                                                                             |

|                                                                                             |
| isLoading.value = true;                                                                     |
|                                                                                             |
| try {                                                                                       |
|                                                                                             |
| const res = await analyticsApi.getDashboard();                                              |
|                                                                                             |
| dashboard.value = res.data;                                                                 |
|                                                                                             |

|                                                                                             |
| } finally {                                                                                 |
|                                                                                             |
| isLoading.value = false;                                                                    |
|                                                                                             |
| }                                                                                           |
|                                                                                             |
| }                                                                                           |
